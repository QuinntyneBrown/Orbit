# L2 - Detailed Requirements

> **Traceability note:** Every acceptance test written against these requirements must include a comment header in the following format:
> ```
> # Acceptance Test
> # Traces to: L2-XXX[, L2-YYY]
> # Description: <one-line summary>
> ```
>
> **Architecture note:** All structured tracking data is stored in a local SQLite database (`data/orbit.db`). Requirements that previously referenced flat files (`data/pipeline.md`, `data/scan-history.tsv`, `content/evaluations/`, `interview-prep/story-bank.md`, `docs/recruiter-vendor-list.md`) now reference database tables. Resume and outreach message content remains as editable files.

## L2-000: Database Schema and Migrations
**Traces to:** L1-003, L1-004, L1-006, L1-010, L1-011

The system must maintain a SQLite database at `data/orbit.db` as the single source of truth for all operational tracking state. The schema must be defined in `db/schema.sql` and managed through numbered migration scripts in `db/migrations/`.

**Acceptance Criteria:**
1. Given the database does not exist, when any script that requires it is run for the first time, then the database must be created automatically by running all pending migrations in order.
2. Given the database exists at a prior schema version, when the application starts, then pending migrations must be applied automatically before any data operation proceeds.
3. Given `db/schema.sql` is read, then it must define tables for: `pipeline_entries`, `offer_evaluations`, `job_listings`, `scan_runs`, `compensation_estimates`, `recruiter_contacts`, `target_accounts`, `interview_stories`, `outreach_records`, and `schema_migrations`.
4. Given any table definition is reviewed, when CHECK constraints are listed, then all enum-like columns must use CHECK constraints to enforce valid values at the database layer.
5. Given the database file exists, when version control status is checked, then `data/orbit.db` must not appear as tracked or untracked (it is gitignored per L2-024).

## L2-001: Base Resume Canonical Structure
**Traces to:** L1-001

The system must maintain exactly two base resume files — a focused (default) variant and a comprehensive variant — as the canonical sources for all tailored content. No tailored variant shall be created without deriving from one of these two files.

**Acceptance Criteria:**
1. Given a new tailored variant is created, when the variant creation command is run, then the output file must be a copy of one of the two base files with no sections removed that are present in the base.
2. Given either base file is edited, when the other base file is checked, then sections shared between them (current role, certifications, contact details) must remain consistent.
3. Given a base file does not exist at its canonical path, when any build script is invoked, then the script must exit with an error message identifying the missing file.

## L2-002: Notes File Template Enforcement
**Traces to:** L1-001

All role and company notes files must conform to a standard template containing sections: Role Snapshot, Baseline Bullets, Additional Bullet Bank, Tailoring Angles, and Keyword Bank. New notes files must use the template. Existing files must be migrated to the template on next edit.

**Acceptance Criteria:**
1. Given a new notes file is created, when the file is opened, then it must contain all five required section headings from the notes template.
2. Given an existing notes file lacks one or more required sections, when the file is opened for editing, then a comment at the top must indicate which sections are missing.
3. Given the notes template does not exist, when a variant is created for a company with no existing notes file, then the script must create a blank notes file using the template structure and warn the user.

## L2-003: Outreach Message Organization
**Traces to:** L1-001

All LinkedIn messages, email drafts, and recruiter follow-up files must reside in the designated outreach directory. No outreach files shall exist in the tailored resume directory. Outreach file metadata (path, version, listing reference) must be recorded in the `outreach_records` database table.

**Acceptance Criteria:**
1. Given outreach files exist in the tailored directory, when a migration script or manual move is performed, then all message files must appear only in the outreach directory and must no longer exist in the tailored directory.
2. Given a new outreach file is generated, when it is saved, then it must be saved following the naming convention `<company>-<role>-<type>` where `<type>` is `linkedin-message`, `email`, or `follow-up`, and a corresponding row must be inserted into `outreach_records`.
3. Given the outreach directory is listed, when filtered for a specific company, then all outreach artifacts for that company must appear in one place.

## L2-004: Markdown to Word Conversion
**Traces to:** L1-002

The build script must convert any Markdown resume file to a Word `.docx` file using Pandoc and a reference style template. The output must be written to the appropriate output directory based on whether the input is a tailored or base variant.

**Acceptance Criteria:**
1. Given a valid Markdown resume file exists, when the build script is run with that file as input, then a `.docx` file must be created at the correct output path within 30 seconds.
2. Given the PDF flag is passed to the build script, when the script runs, then both a `.docx` and a PDF must be produced.
3. Given the reference template does not exist, when the build script is run, then the script must exit with an error identifying the missing template.
4. Given a Markdown file contains standard resume sections (Contact, Summary, Experience, Skills), when converted, then the output `.docx` must preserve all section headings and bullet structure without data loss.

## L2-005: Playwright-Based PDF Generation
**Traces to:** L1-002

The system must provide a PDF generation script that produces a PDF from a Markdown resume using Playwright (HTML render → PDF) without requiring Microsoft Word. A corresponding HTML template must exist that mirrors the `.docx` visual style.

**Acceptance Criteria:**
1. Given the PDF generation script is run with a Markdown resume file as input, when Playwright is available, then a PDF must be produced in the exports directory within 60 seconds.
2. Given Microsoft Word is not installed, when the PDF generation script is run, then the PDF must still be generated successfully.
3. Given the generated PDF, when opened, then fonts, heading hierarchy, bullet structure, and contact information must be visually equivalent to the `.docx` output.
4. Given the HTML template does not exist, when the PDF generation script is run, then the script must exit with a clear error identifying the missing template.
5. Given a resume file path is passed, when the output PDF is created, then it must be named after the input file basename and written to the exports directory.

## L2-006: Base Resume Sync Verification
**Traces to:** L1-002

A sync verification script must compare key sections (current role, certifications, dates, contact details) between the focused and comprehensive base resume files and warn when they diverge beyond expected differences — specifically when the focused resume contains information that contradicts the comprehensive resume rather than simply omitting it.

**Acceptance Criteria:**
1. Given both base files are in sync on key sections, when the sync verification script is run, then the script must exit with a success message and no warnings.
2. Given the current role title differs between the two base files, when the sync verification script is run, then the script must output a warning identifying the diverging field.
3. Given the script detects divergence, when it exits, then the exit code must be non-zero so it can fail in CI or pre-commit hooks.
4. Given a key section is present in the focused resume but absent in the comprehensive resume, when the script runs, then the script must flag this as an error (content should flow from comprehensive → focused, not the reverse).

## L2-007: Application Pipeline Tracker
**Traces to:** L1-003

The system must maintain a `pipeline_entries` table in the SQLite database tracking all roles applied to. Required columns must include: sequence number, applied date, company, role, source, status, rate, pdf path, evaluation id (FK to `offer_evaluations`), and notes. The table must be updated after every tailored resume build.

**Acceptance Criteria:**
1. Given a tailored resume is built, when the build completes, then the user must be prompted (or the script must instruct) to log the entry to the pipeline via a DB insert.
2. Given the database is accessible, when a new pipeline entry is inserted, then all required columns must be non-null (except rate and notes); the database must reject inserts with missing required fields.
3. Given a role's status changes (e.g., Applied → Interview), when the pipeline is updated via `UPDATE pipeline_entries SET status = ?, updated_at = ? WHERE id = ?`, then only the status and updated_at columns must change; other columns must remain intact.
4. Given the pipeline is queried for all entries with a specific status, when `SELECT * FROM pipeline_entries WHERE status = ?` is executed, then only rows with that exact status must be returned.

## L2-008: Canonical Application Status Model
**Traces to:** L1-003

Valid application status values must be enforced by a CHECK constraint on the `pipeline_entries.status` column. Valid states are: `Evaluated`, `Applied`, `Responded`, `Interview`, `Offer`, `Rejected`, `Discarded`, `SKIP`. No other status values are permitted.

**Acceptance Criteria:**
1. Given the `pipeline_entries` table has a CHECK constraint on `status`, when an INSERT or UPDATE with a non-canonical status (e.g., `Pending`) is attempted, then the database must reject the operation with a constraint violation error.
2. Given an attempt to insert a status value with mixed case (e.g., `applied`), when the constraint is evaluated, then the insert must be rejected (status is case-sensitive).
3. Given the schema is reviewed, when CHECK constraints are listed, then the status column must enumerate all eight valid values explicitly in the constraint definition.

## L2-009: Offer Evaluation Storage
**Traces to:** L1-004

Each offer evaluation must be stored as a row in the `offer_evaluations` table immediately after completion, with all five scored dimensions as individual columns. Re-evaluations of the same role must create a new row with an incremented version number; the previous row must be linked via the `superseded_by` foreign key rather than deleted. The corresponding `pipeline_entries` row must be updated with the new evaluation's id.

**Acceptance Criteria:**
1. Given an offer evaluation is completed, when the record is saved, then a row must be inserted into `offer_evaluations` with all five dimension ratings, computed score, label, and recommended action populated.
2. Given a completed evaluation row is read, when `recommended_action` is checked, then it must be one of: `Tailor`, `Watch`, `Skip`.
3. Given a role earns a computed score below 3.0, when the evaluation is saved, then `recommended_action` must be `Skip` enforced by a CHECK constraint.
4. Given an evaluation already exists for the same company+role, when a new evaluation is saved, then the new row's `version` must be incremented and the old row's `superseded_by` must be set to the new row's id.

## L2-010: Weighted Offer Scoring
**Traces to:** L1-004

Every offer evaluation must produce a numeric score from 0.0 to 5.0 based on five weighted dimensions: Technical match (35%), Seniority alignment (25%), Archetype fit (20%), Compensation fairness (10%), Market demand (10%). A score of 4.5 or above must trigger a recommendation to begin tailoring immediately.

**Acceptance Criteria:**
1. Given an evaluation covers all five dimensions with individual ratings (A/B/C/Skip), when the final score is computed, then it must equal the weighted sum of the five numeric dimension values.
2. Given a computed score of 4.5 or above, when the evaluation is saved to the database, then `recommended_action` must be `Tailor` and `label` must be `Priority`.
3. Given a computed score below 3.0, when the evaluation is saved, then `recommended_action` must be `Skip` and `label` must be `Low Fit`.
4. Given any of the five dimension columns is NULL when the score is computed, then the computation must raise an error identifying the missing dimension; the record must not be inserted with a NULL score.

## L2-011: Job Board Search
**Traces to:** L1-005

The job search module must search at minimum the following categories of job boards: major general-purpose boards (e.g., LinkedIn, Indeed, Glassdoor) and remote-specific boards. Searches must use the candidate's configured target keywords and location preferences as defined in `config/profile.yml`.

**Acceptance Criteria:**
1. Given the module is run without additional flags, when results are returned, then listings from at least two distinct job boards must be included.
2. Given a search runs across boards, when results are persisted, then each `job_listings` row must have: title, company, source, posted_date, rate, url populated.
3. Given a keyword not relevant to the profile is searched, when results are returned, then a warning must indicate zero matches for that keyword rather than silently omitting it.
4. Given results are returned, when displayed, then listings with `is_stale = 1` (posted > 30 days ago) must be visually distinguished (e.g., marked as `[Stale]`).

## L2-012: Company Portal Scanner
**Traces to:** L1-005

The job search module must support a portal scan mode that queries the career pages of companies in the `target_accounts` table and checks for open roles matching the candidate's keywords. Supported ATS platforms must include: Greenhouse, Ashby, Lever, Wellfound, and Workable.

**Acceptance Criteria:**
1. Given portal scan mode is invoked, when the `target_accounts` table is queried, then the module must attempt to scan every company with a non-null `career_page_url`.
2. Given a company career page is scanned via a supported ATS, when matching roles are found, then each must be inserted into `job_listings` with title, company, ats_type, url, and posted_date.
3. Given a company's career page returns a non-200 HTTP status, when the scan completes, then the failure must be recorded and surfaced in the run's export under a "Failed to scan" section.
4. Given a company in `target_accounts` has a NULL `career_page_url`, when scanning, then that company must be reported in a "No portal configured" section.

## L2-013: Recruiter Board Search
**Traces to:** L1-005

The job search module must support a recruiter board search mode that searches the opportunity pages of staffing vendors in the `recruiter_contacts` table with `priority_tier = 'High'`. Results from high-priority recruiters must be flagged distinctly.

**Acceptance Criteria:**
1. Given recruiter board search mode is invoked, when `recruiter_contacts` is queried, then all vendors with `priority_tier = 'High'` must be searched.
2. Given a recruiter board is searched and a matching role is found, then the `job_listings` row must have `is_priority_recruiter = 1` and the recruiter firm name must appear in the export.
3. Given a recruiter board URL is unavailable, when the scan completes, then the failed vendor must be reported in a "Failed to scan" section rather than silently skipping.

## L2-014: Search Result Persistence
**Traces to:** L1-006

After each job search run, all results must be persisted to the `job_listings` table with deduplication on `(company, title)`. Each run must create a row in `scan_runs`. On subsequent runs, listings where the `(company, title)` combination already exists must be updated to `status = 'Seen'` rather than inserted as new rows.

**Acceptance Criteria:**
1. Given a search run completes, when results are written to the database, then a row must be inserted in `scan_runs` and each result must exist as a row in `job_listings` with `status = 'New'` and `first_seen_date` set to today.
2. Given a search run finds a listing whose `(company, title)` already exists in `job_listings`, when the upsert runs, then the existing row's `status` must be set to `Seen` and `last_seen_date` updated; no duplicate row must be created.
3. Given a `job_listings` row has `status = 'Applied'` (set by the user), when a subsequent search finds the same listing, then the `status` must remain `Applied` and must not be overwritten by the scan.
4. Given the database does not exist when a search run completes, then the database must be initialized (L2-000 AC1) and then results written.

## L2-015: Search Run Export
**Traces to:** L1-006, L1-014

After each search run, a human-readable Markdown export must be generated from the database and written to `data/search-results/<YYYY-MM-DD>.md`. The system must retain exports for the most recent N runs (configurable, default 8). The top of each export must include a diff summary computed by comparing the current `scan_runs` entry against the previous one.

**Acceptance Criteria:**
1. Given a search run completes, when the export is generated, then the file must appear in `data/search-results/` named with today's ISO date, and all listing data must be read from `job_listings WHERE scan_run_id = <current>`.
2. Given more than N exports exist on disk, when a new export is written, then the oldest file must be deleted so that exactly N remain. N is read from `config/search-settings.json` key `resultHistoryWindow` (default 8).
3. Given the export file is read, when the first section is parsed, then it must contain a diff block: count of new listings (`status = 'New'` in current run), count of listings absent from current run vs. previous, and count of status-changed listings — all computed by SQL query against `scan_runs` and `job_listings`.
4. Given it is the first-ever search run (no prior `scan_runs` row), when the export is written, then the diff block must read "First run — no prior results to compare."

## L2-016: Compensation Research for Unrated Postings
**Traces to:** L1-007

When a `job_listings` row has a null or blank `rate` field, the compensation research module must perform a targeted research step and store the result in the `compensation_estimates` table. Estimates must not overwrite an explicit rate. Results older than 30 days must be re-researched on the next run.

**Acceptance Criteria:**
1. Given a `job_listings` row has no rate value, when the module processes that listing, then a row must be inserted or updated in `compensation_estimates` for that `listing_id` with at least one cited source.
2. Given a `compensation_estimates` row is read, when fields are checked, then `range_low`, `range_high`, `confidence` (High/Medium/Low), and `source` must all be non-null.
3. Given a `job_listings` row has an explicit `rate` value, when compensation research runs, then no `compensation_estimates` row must be inserted or updated for that listing.
4. Given all reachable salary sources return no data for a given role and location, when the research step completes, then the `compensation_estimates` row must record `range_low = NULL`, `range_high = NULL`, and `source = 'No data found'`.

## L2-017: Role Archetype Classification
**Traces to:** L1-008

Every `job_listings` row must have a non-null `archetype` value after classification runs. The archetype must be surfaced in exports, evaluation records, and pipeline entries.

**Acceptance Criteria:**
1. Given a `job_listings` row is processed, when archetype classification runs, then the `archetype` column must be set to exactly one of the five defined values; NULL is not permitted.
2. Given a listing is classified as `Enterprise Contract`, when a tailored resume is recommended, then the recommendation must reference the focused base resume as the preferred starting point.
3. Given a listing is classified as `Government / Public Sector`, when displayed, then the output must include a flag reminding the candidate to reference any active security clearance in the tailored resume.
4. Given a listing's archetype cannot be determined, when classification runs, then `archetype` must be set to `Enterprise Contract` and `archetype_inferred` must be set to `1`.

## L2-018: LinkedIn Outreach Generation
**Traces to:** L1-009

The job search module must support an outreach mode that, for each listing with `auto_score >= 4.5`, generates a personalized LinkedIn message draft and saves it to `content/outreach/`. A row must be inserted into `outreach_records` linking the file to its `job_listings` row. Version collision must be resolved via the `version` column in `outreach_records`.

**Acceptance Criteria:**
1. Given outreach mode is active and a `job_listings` row has `auto_score >= 4.5`, when the module completes, then a file must exist in `content/outreach/` and a row must exist in `outreach_records` for that listing.
2. Given an outreach file is generated, when its content is read, then it must contain: a personalized opening referencing the specific role or company, a 2–3 sentence value proposition drawn from the candidate profile, and a clear call to action.
3. Given a listing has `auto_score < 4.5`, when outreach mode is active, then no file must be created and no `outreach_records` row must be inserted for that listing.
4. Given an `outreach_records` row already exists for the same `listing_id`, when outreach mode runs again, then a new file must be created with an incremented version suffix and a new `outreach_records` row inserted with `version` incremented.

## L2-019: Interview Story Bank
**Traces to:** L1-010

The system must maintain behavioral interview stories in the `interview_stories` table. Each story must include all four STAR components plus a Reflection field, and at least one keyword tag (stored as a JSON array in the `keywords` column). Stories must never be updated or deleted — new versions are appended as new rows.

**Acceptance Criteria:**
1. Given a new story is added, when the INSERT is executed, then the row must include non-null values for `situation`, `task`, `action`, `result`, `reflection`, and at least one entry in the `keywords` JSON array.
2. Given a role in `pipeline_entries` reaches `status = 'Interview'`, when the story surfacing query runs, then at least two stories must be returned based on keyword overlap between `interview_stories.keywords` and the role's job description keywords.
3. Given `SELECT COUNT(*) FROM interview_stories` returns fewer than 3, when an evaluation runs for an Interview-stage role, then the output must include a warning prompting the candidate to add stories.
4. Given a story is added, when the INSERT completes, then existing rows in `interview_stories` must remain unchanged (no UPDATE or DELETE on the table is permitted by application code).

## L2-020: Recruiter Relationship Tracking
**Traces to:** L1-011

Recruiter and staffing vendor data must be stored in the `recruiter_contacts` table with columns: `firm_name`, `priority_tier`, `contact_name`, `contact_linkedin`, `opportunity_page_url`, `last_contacted_date`, `engagement_status`, and `notes`. High-priority contacts not updated in 90+ days must be surfaced automatically.

**Acceptance Criteria:**
1. Given the `recruiter_contacts` table is queried, when all rows are returned, then every row must have non-null values for `firm_name`, `priority_tier`, and `last_contacted_date`.
2. Given a recruiter is contacted about a specific role, when `UPDATE recruiter_contacts SET last_contacted_date = ?, engagement_status = ? WHERE id = ?` is run, then those two columns must reflect the new values.
3. Given `SELECT * FROM recruiter_contacts WHERE priority_tier = 'High' AND last_contacted_date <= date('now', '-90 days')` returns results, when a follow-up report runs, then those contacts must be listed in a "Due for follow-up" section.
4. Given the `target_accounts` table references a company that also exists in `recruiter_contacts`, when a cross-reference check runs, then `target_accounts.recruiter_contact_id` must be set to the matching `recruiter_contacts.id`.

## L2-021: Tailored Resume Variant Creation
**Traces to:** L1-012

The variant creation script must create a tailored resume file in the tailored content directory by copying the specified base file (defaulting to the focused base resume), naming it according to the target role slug, and optionally creating a corresponding notes file from the notes template.

**Acceptance Criteria:**
1. Given the variant creation script is run with a role name argument, when the script completes, then a tailored resume file must exist in the tailored directory as a copy of the base file.
2. Given the notes flag is passed to the variant creation script, when the script completes, then a notes file must exist in the notes directory as a copy of the notes template.
3. Given a tailored file with the specified name already exists, when the variant creation script is run, then the script must prompt for confirmation before overwriting the existing file.
4. Given the base file argument is omitted, when the variant creation script runs, then it must default to the focused base resume without requiring the user to specify it.

## L2-022: Batch Tailoring
**Traces to:** L1-012

The system must provide a batch tailoring script that accepts a list of role names, runs the variant creation and tailoring steps for each role in parallel, and reports completion status for each job.

**Acceptance Criteria:**
1. Given the batch tailoring script is run with a list of three role names, when the script completes, then three tailored Markdown files and three `.docx` files must exist in the correct output directories.
2. Given batch jobs run in parallel, when one job fails (e.g., invalid role name), then the failed job must be reported in an error summary without blocking other jobs.
3. Given a batch of five or more roles, when the script runs, then no more than four jobs must execute concurrently to prevent resource exhaustion.
4. Given all jobs complete, when the summary is printed, then it must show: succeeded count, failed count, and total elapsed time.

## L2-023: Session Onboarding and Environment Validation
**Traces to:** L1-013

The job search module must verify at startup: (a) the base resume file exists, (b) the file was last modified within 90 days, (c) the database is reachable and all migrations have been applied, and (d) the module reads the actual file at runtime rather than using a static profile summary.

**Acceptance Criteria:**
1. Given the base resume file does not exist, when the module starts, then it must exit with an error identifying the missing file and indicating it cannot proceed.
2. Given the base resume exists but was last modified more than 90 days ago, when the module starts, then it must display a warning and prompt the user to confirm before continuing.
3. Given the database exists but has unapplied migrations, when the module starts, then it must apply all pending migrations before proceeding and log each migration applied.
4. Given the database does not exist, when the module starts, then it must create and initialize the database (running all migrations) before any search or evaluation logic executes.
5. Given the base resume passes all checks and the database is ready, when the module starts, then no onboarding warnings must be shown and the module must proceed immediately.

## L2-024: Sensitive Data Exclusion from Version Control
**Traces to:** L1-015

The database file and all content files containing personal data must be excluded from version control. `data/orbit.db` must be gitignored. Content directories containing personal documents must also be excluded.

**Acceptance Criteria:**
1. Given `data/orbit.db` exists locally, when `git status` is run, then the file must not appear as tracked or untracked.
2. Given `content/tailored/`, `content/outreach/`, and `content/notes/` contain files, when `git status` is run, then no files from those directories must appear as tracked or untracked.
3. Given the `.gitignore` file is reviewed, when the list of exclusions is compared to the required set (`data/orbit.db`, `content/tailored/`, `content/outreach/`, `data/search-results/`, `config/profile.yml`, `resumes/`, `exports/`), then all required exclusions must be present.
4. Given a file is accidentally staged despite being excluded, when version control status is checked, then the file must show as excluded if the force-add flag was not used.

## L2-025: Offer Evaluation Versioning
**Traces to:** L1-004

Each re-evaluation of the same role must produce a new `offer_evaluations` row rather than overwriting the previous one. The previous row must be marked as superseded. The `pipeline_entries.eval_id` must always reference the most recent evaluation for that role.

**Acceptance Criteria:**
1. Given an offer evaluation is completed for a role, when the row is saved, then `offer_evaluations.version` must be `1` for first evaluations and increment by 1 for each subsequent evaluation of the same company+role.
2. Given a new evaluation is saved for a previously evaluated role, when the old row is checked, then `offer_evaluations.superseded_by` must be set to the id of the new row.
3. Given `pipeline_entries.eval_id` is read for a role, when the referenced `offer_evaluations` row is fetched, then it must be the most recent version (i.e., `superseded_by IS NULL`).

## L2-026: Search Output Format Standardization
**Traces to:** L1-005, L1-014

The job search module must produce human-readable Markdown exports generated from database queries. The export format must include a metadata header block, a diff summary, and a per-listing block for each result in the current scan run.

**Acceptance Criteria:**
1. Given the module completes a search, when the export is written, then the file must begin with a YAML front-matter block containing: `date`, `total_results`, `boards_searched`, `new_listings`, and `seen_listings` — all values read from the `scan_runs` row for this run.
2. Given the export is parsed programmatically, when each listing block is read, then it must contain all required fields (`title`, `company`, `source`, `date`, `rate`, `url`, `archetype`, `score`) sourced from `job_listings` rows for the current `scan_run_id`.
3. Given two consecutive exports are diffed, then the diff must be interpretable by the diff summary generator (L2-015 AC3) without additional preprocessing.
