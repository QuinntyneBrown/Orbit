# L2 - Detailed Requirements

> **Traceability note:** Every acceptance test written against these requirements must include a comment header in the following format:
> ```
> # Acceptance Test
> # Traces to: L2-XXX[, L2-YYY]
> # Description: <one-line summary>
> ```

---

## L2-001: Base Resume Canonical Structure
**Traces to:** L1-001

The system must maintain exactly two base resume files — a focused (default) variant and a comprehensive variant — as the canonical sources for all tailored content. No tailored variant shall be created without deriving from one of these two files.

**Acceptance Criteria:**
1. Given a new tailored variant is created, when the variant creation command is run, then the output file must be a copy of one of the two base files with no sections removed that are present in the base.
2. Given either base file is edited, when the other base file is checked, then sections shared between them (current role, certifications, contact details) must remain consistent.
3. Given a base file does not exist at its canonical path, when any build script is invoked, then the script must exit with an error message identifying the missing file.

---

## L2-002: Notes File Template Enforcement
**Traces to:** L1-001

All role and company notes files must conform to a standard template containing sections: Role Snapshot, Baseline Bullets, Additional Bullet Bank, Tailoring Angles, and Keyword Bank. New notes files must use the template. Existing files must be migrated to the template on next edit.

**Acceptance Criteria:**
1. Given a new notes file is created, when the file is opened, then it must contain all five required section headings from the notes template.
2. Given an existing notes file lacks one or more required sections, when the file is opened for editing, then a comment at the top must indicate which sections are missing.
3. Given the notes template does not exist, when a variant is created for a company with no existing notes file, then the script must create a blank notes file using the template structure and warn the user.

---

## L2-003: Outreach Message Organization
**Traces to:** L1-001

All LinkedIn messages, email drafts, and recruiter follow-up files must reside in the designated outreach directory. No outreach files shall exist in the tailored resume directory. Misplaced outreach files must be migrated to the correct location.

**Acceptance Criteria:**
1. Given outreach files exist in the tailored directory, when a migration script or manual move is performed, then all message files must appear only in the outreach directory and must no longer exist in the tailored directory.
2. Given a new outreach file is generated, when it is saved, then it must be saved following the naming convention `<company>-<role>-<type>` where `<type>` is `linkedin-message`, `email`, or `follow-up`.
3. Given the outreach directory is listed, when filtered for a specific company, then all outreach artifacts for that company must appear in one place.

---

## L2-004: Markdown to Word Conversion
**Traces to:** L1-002

The build script must convert any Markdown resume file to a Word `.docx` file using Pandoc and a reference style template. The output must be written to the appropriate output directory based on whether the input is a tailored or base variant.

**Acceptance Criteria:**
1. Given a valid Markdown resume file exists, when the build script is run with that file as input, then a `.docx` file must be created at the correct output path within 30 seconds.
2. Given the PDF flag is passed to the build script, when the script runs, then both a `.docx` and a PDF must be produced.
3. Given the reference template does not exist, when the build script is run, then the script must exit with an error identifying the missing template.
4. Given a Markdown file contains standard resume sections (Contact, Summary, Experience, Skills), when converted, then the output `.docx` must preserve all section headings and bullet structure without data loss.

---

## L2-005: Playwright-Based PDF Generation
**Traces to:** L1-002

The system must provide a PDF generation script that produces a PDF from a Markdown resume using Playwright (HTML render → PDF) without requiring Microsoft Word. A corresponding HTML template must exist that mirrors the `.docx` visual style.

**Acceptance Criteria:**
1. Given the PDF generation script is run with a Markdown resume file as input, when Playwright is available, then a PDF must be produced in the exports directory within 60 seconds.
2. Given Microsoft Word is not installed, when the PDF generation script is run, then the PDF must still be generated successfully.
3. Given the generated PDF, when opened, then fonts, heading hierarchy, bullet structure, and contact information must be visually equivalent to the `.docx` output.
4. Given the HTML template does not exist, when the PDF generation script is run, then the script must exit with a clear error identifying the missing template.
5. Given a resume file path is passed, when the output PDF is created, then it must be named after the input file basename and written to the exports directory.

## L2-006: Base Resume Sync Verification
**Traces to:** L1-002

A sync verification script must compare key sections (current role, certifications, dates, contact details) between the focused and comprehensive base resume files and warn when they diverge beyond expected differences — specifically when the focused resume contains information that contradicts the comprehensive resume rather than simply omitting it.

**Acceptance Criteria:**
1. Given both base files are in sync on key sections, when the sync verification script is run, then the script must exit with a success message and no warnings.
2. Given the current role title differs between the two base files, when the sync verification script is run, then the script must output a warning identifying the diverging field.
3. Given the script detects divergence, when it exits, then the exit code must be non-zero so it can fail in CI or pre-commit hooks.
4. Given a key section is present in the focused resume but absent in the comprehensive resume, when the script runs, then the script must flag this as an error (content should flow from comprehensive → focused, not the reverse).

## L2-007: Application Pipeline Tracker
**Traces to:** L1-003

The system must maintain a pipeline tracking file as a structured table tracking all roles applied to. Required fields must include: sequence number, date, company, role, source, status, compensation, document links, and notes. The file must be updated after every tailored resume build.

**Acceptance Criteria:**
1. Given a tailored resume is built, when the build completes, then the user must be prompted (or the script must instruct) to log the entry to the pipeline tracker.
2. Given the pipeline tracker exists, when a new entry is added, then all required fields must be populated; compensation and notes may be empty but all other fields must have values.
3. Given a role's status changes (e.g., Applied → Interview), when the pipeline is updated, then only the status field of that row must change; other fields must remain intact.
4. Given the pipeline is queried for all entries with a specific status, when filtered, then only rows with that exact status must be returned.

---

## L2-008: Canonical Application Status Model
**Traces to:** L1-003

Valid application status values must be defined in a states configuration file and enforced across all pipeline entries. Valid states are: `Evaluated`, `Applied`, `Responded`, `Interview`, `Offer`, `Rejected`, `Discarded`, `SKIP`. No other status values are permitted in the pipeline tracker.

**Acceptance Criteria:**
1. Given the states configuration lists the eight canonical states, when a pipeline entry is created, then the status value must exactly match one of the eight defined states (case-sensitive).
2. Given an entry with a non-canonical status (e.g., `Pending`, `On Hold`) exists in the pipeline, when the pipeline is validated, then the validation must report the invalid status and the row identifier.
3. Given a status value contains markdown formatting or embedded dates, when the pipeline is validated, then those entries must be flagged as invalid.

## L2-009: Offer Evaluation Template
**Traces to:** L1-004

The system must provide an offer evaluation template containing a structured evaluation covering: role fit vs. candidate profile, rate vs. target, remote/hybrid terms, domain alignment, company stability, contract length, interview likelihood, and growth potential. Each dimension must include a scoring field (A/B/C/Skip) and a notes field. A dedicated evaluations folder must exist for per-role evaluation files.

**Acceptance Criteria:**
1. Given the offer evaluation template exists, when a new evaluation is started, then the evaluator must copy the template to the evaluations directory using the naming convention `<company>-<role>-eval`.
2. Given a completed evaluation file, when all scoring fields are populated, then the file must include a final recommended action field with one of: `Tailor`, `Skip`, `Watch`.
3. Given a role earns an overall score of `C` or `Skip`, when the evaluation is complete, then the recommended action must be `Skip` and no tailored resume must be created for that role.
4. Given the evaluation template is updated, when existing evaluations are opened, then they must still be valid (the template must be additive only; fields must not be removed).

---

## L2-010: Weighted Offer Scoring
**Traces to:** L1-004

Every offer evaluation must produce a numeric score from 1.0 to 5.0 based on five weighted dimensions: Technical match (35%), Seniority alignment (25%), Archetype fit (20%), Compensation fairness (10%), Market demand (10%). A score of 4.5 or above must trigger a recommendation to begin tailoring immediately.

**Acceptance Criteria:**
1. Given an evaluation covers all five dimensions with individual scores, when the final score is computed, then it must equal the weighted sum of the five dimension scores normalized to a 1–5 scale.
2. Given a computed score of 4.5 or above, when the evaluation is saved, then the recommended action must be `Tailor` and the evaluation must note "Priority — begin tailoring immediately."
3. Given a computed score below 3.0, when the evaluation is saved, then the recommended action must be `Skip`.
4. Given a dimension score is missing, when the final score is computed, then the computation must fail with an error identifying the missing dimension rather than silently defaulting to zero.

## L2-011: Job Board Search
**Traces to:** L1-005

The job search module must search at minimum the following categories of job boards: major general-purpose boards (e.g., LinkedIn, Indeed, Glassdoor) and remote-specific boards. Searches must use the candidate's configured target keywords and location preferences as defined in the candidate profile.

**Acceptance Criteria:**
1. Given the module is run without additional flags, when results are returned, then listings from at least two distinct job boards must be included.
2. Given a search runs across boards, when results are displayed, then each listing must show: Title, Company, Board Source, Posted Date, Rate (or "Rate not listed"), and URL.
3. Given a keyword not relevant to the profile is searched, when results are returned, then a warning must indicate zero matches for that keyword rather than silently omitting the keyword from the search.
4. Given results are returned, when displayed, then listings older than 30 days must be visually distinguished (e.g., marked as `[Stale]`).

## L2-012: Company Portal Scanner
**Traces to:** L1-005

The job search module must support a portal scan mode that navigates directly to the career pages of companies listed in the target account list and checks for open roles matching the candidate's keywords. Supported ATS platforms must include: Greenhouse, Ashby, Lever, Wellfound, and Workable.

**Acceptance Criteria:**
1. Given portal scan mode is invoked, when the target account list is read, then the module must attempt to scan every company with a non-empty careers URL field.
2. Given a company career page is scanned via a supported ATS API, when matching roles are found, then each must be returned with: Title, Company, ATS Platform, URL, and Posted Date.
3. Given a company's career page returns a non-200 HTTP status, when the scan completes, then the company must be listed in a "Failed to scan" section with the error code rather than silently omitted.
4. Given portal scan mode is run and a company has no careers URL configured, when scanning, then that company must be listed in a "No portal configured" section.

---

## L2-013: Recruiter Board Search
**Traces to:** L1-005

The job search module must support a recruiter board search mode that searches the opportunities pages of priority staffing vendors listed in the recruiter vendor list. Results sourced from known high-priority recruiters must be flagged distinctly in output.

**Acceptance Criteria:**
1. Given recruiter board search mode is invoked, when the recruiter vendor list is read, then all vendors with a `Priority: High` designation must be searched.
2. Given a recruiter board is searched, when a matching role is found, then the listing must include a `[Priority Recruiter]` flag in the output alongside the recruiter firm name.
3. Given a recruiter board URL is unavailable, when the scan completes, then the failed vendor must appear in a "Failed to scan" section rather than silently skipping.

---

## L2-014: Search Result Persistence
**Traces to:** L1-006

After each job search run, the full result set must be appended to a persistent scan history file with columns: Date, Title, Company, Board, URL, Status (New/Seen/Applied). On subsequent runs, listings where the Company+Title combination already exists in the history must be marked `Seen` rather than `New`.

**Acceptance Criteria:**
1. Given a search run completes, when results are written to the scan history file, then each listing must appear as a new row with all required columns populated and `Status = New`.
2. Given a search run finds a listing whose Company+Title already exists in the scan history, when the result is written, then the `Status` must be `Seen` and the existing row must not be duplicated.
3. Given a listing in the scan history has `Status = Applied` (set by the user), when a subsequent search finds the same listing, then the `Status` must remain `Applied` and must not be overwritten by the scan.
4. Given the scan history file does not exist, when a search run completes, then the file must be created with the correct header row before any results are written.

## L2-015: Dated Search Result Export
**Traces to:** L1-006, L1-014

After each search run, a human-readable results file must be written to the search results directory using an ISO date filename. The system must retain only the most recent 8 run files (rolling window). The top of each file must include a diff summary: net new listings, removed listings, and status-changed listings compared to the previous run.

**Acceptance Criteria:**
1. Given a search run completes, when the result file is written, then it must appear in the search results directory named with today's ISO date.
2. Given more than 8 result files exist in the search results directory, when a new file is written, then the oldest file must be deleted so that exactly 8 remain.
3. Given a result file is opened, when the first section is read, then it must contain a diff block showing: count of net new listings, count of listings no longer found, and count of listings with changed status.
4. Given it is the first-ever search run (no prior file exists), when the result file is written, then the diff block must read "First run — no prior results to compare."

## L2-016: Compensation Research for Unrated Postings
**Traces to:** L1-007

When a job listing does not include a rate or salary, the job search module must perform a targeted compensation research step and report a rate estimate alongside that listing. Research must draw on publicly visible data points (e.g., Glassdoor, Indeed salary estimates, comparable current listings).

**Acceptance Criteria:**
1. Given a listing has no salary or rate field, when the module processes that listing, then a `Rate estimate` field must appear in the output with at least one cited data source.
2. Given a rate estimate is computed, when displayed, then it must include a range (not a single number) and a confidence qualifier: `High`, `Medium`, or `Low`.
3. Given a listing includes an explicit rate, when processed, then no compensation research step must be performed for that listing.
4. Given all reachable salary sources return no data for a given role and location, when the research step completes, then the `Rate estimate` field must display "No data found" rather than omitting the field.

## L2-017: Role Archetype Classification
**Traces to:** L1-008

Every job listing processed by the module must be classified into exactly one of five archetypes: `Enterprise Contract`, `Product Company`, `Consulting Firm`, `AI / Innovation`, `Government / Public Sector`. The archetype must be surfaced in search results, evaluation reports, and pipeline tracker entries.

**Acceptance Criteria:**
1. Given a job listing is processed, when archetype classification runs, then exactly one archetype must be assigned; no listing may carry multiple archetypes or no archetype.
2. Given a listing is classified as `Enterprise Contract`, when a tailored resume is recommended, then the recommendation must reference the focused base resume as the preferred starting point.
3. Given a listing is classified as `Government / Public Sector`, when displayed, then the output must include a flag reminding the user to reference any active security clearance in the tailored resume.
4. Given a listing's archetype cannot be determined from available text, when classification runs, then the archetype must default to `Enterprise Contract` and the listing must be flagged as `[Archetype: inferred]`.

## L2-018: LinkedIn Outreach Generation
**Traces to:** L1-009

The job search module must support an outreach mode that, for each strong match (score ≥ 4.5) listing, generates a personalized LinkedIn message draft and saves it to the outreach directory using the naming convention `<company>-<role>-linkedin-message`. Drafts must draw on the offer evaluation context and mirror the style of existing message examples in the outreach directory.

**Acceptance Criteria:**
1. Given outreach mode is active and a listing scores ≥ 4.5, when the module completes, then a LinkedIn message file must exist in the outreach directory for that company and role.
2. Given an outreach file is generated, when its content is read, then it must contain: a personalized opening referencing the specific role or company, a 2–3 sentence value proposition drawn from the candidate profile, and a clear call to action.
3. Given a listing scores below 4.5, when outreach mode is active, then no outreach file must be created for that listing.
4. Given an outreach file already exists for the same company+role, when outreach mode runs again, then the existing file must not be overwritten; a new file must be created with a `-v2` suffix.

## L2-019: Interview Story Bank
**Traces to:** L1-010

The system must maintain a cumulative bank of behavioral interview stories in STAR+Reflection format. Each story must be tagged with: situation context, skills demonstrated, and mappable job description keywords. The story bank must be referenced during offer evaluation for roles reaching the interview stage.

**Acceptance Criteria:**
1. Given the story bank file exists, when a new story is added, then it must include all four STAR components (Situation, Task, Action, Result) plus a Reflection field, and at least one keyword tag.
2. Given an offer evaluation is completed for a role in the `Interview` state, when the evaluation references the story bank, then at least two relevant stories must be surfaced based on keyword overlap with the job description.
3. Given the story bank contains fewer than three stories, when the module runs an evaluation, then the output must include a warning prompting the user to add stories before the interview.
4. Given a story is added during an evaluation, when saved to the story bank, then it must be appended at the end of the file without modifying existing entries.

## L2-020: Recruiter Relationship Tracking
**Traces to:** L1-011

The recruiter vendor list must include columns for: Firm Name, Priority Tier (High/Medium/Low), Contact Name, Contact LinkedIn, Last Contacted Date, Current Engagement Status, and Notes. The document must be reviewed and updated at least quarterly.

**Acceptance Criteria:**
1. Given the recruiter vendor list is opened, when all columns are checked, then every row must have at minimum: Firm Name, Priority Tier, and Last Contacted Date.
2. Given a recruiter is contacted about a specific role, when the list is updated, then the Last Contacted Date and Current Engagement Status must be updated to reflect the interaction.
3. Given the Last Contacted Date for a High-priority recruiter is more than 90 days ago, when the list is reviewed, then that recruiter must be surfaced in a "Due for follow-up" section.
4. Given the target account list references a company, when cross-checked against the recruiter vendor list, then any firm appearing on both must display a link or cross-reference to the other document.

## L2-021: Tailored Resume Variant Creation
**Traces to:** L1-012

The variant creation script must create a tailored resume file in the tailored content directory by copying the specified base file (defaulting to the focused base resume), naming it according to the target role slug, and optionally creating a corresponding notes file from the notes template.

**Acceptance Criteria:**
1. Given the variant creation script is run with a role name argument, when the script completes, then a tailored resume file must exist in the tailored directory as a copy of the base file.
2. Given the notes flag is passed to the variant creation script, when the script completes, then a notes file must exist in the notes directory as a copy of the notes template.
3. Given a tailored file with the specified name already exists, when the variant creation script is run, then the script must prompt for confirmation before overwriting the existing file.
4. Given the base file argument is omitted, when the variant creation script runs, then it must default to the focused base resume without requiring the user to specify it.

## L2-022: Batch Tailoring
**Traces to:** L1-012

The system must provide a batch tailoring script that accepts a list of role names, runs the variant creation and tailoring steps for each role in parallel, and reports completion status for each job.

**Acceptance Criteria:**
1. Given the batch tailoring script is run with a list of three role names, when the script completes, then three tailored Markdown files and three `.docx` files must exist in the correct output directories.
2. Given batch jobs run in parallel, when one job fails (e.g., invalid role name), then the failed job must be reported in an error summary without blocking other jobs.
3. Given a batch of five or more roles, when the script runs, then no more than four jobs must execute concurrently to prevent resource exhaustion.
4. Given all jobs complete, when the summary is printed, then it must show: succeeded count, failed count, and total elapsed time.

## L2-023: Session Onboarding and Environment Validation
**Traces to:** L1-013

The job search module must verify at startup that: (a) the base resume file exists, (b) the file was last modified within the past 90 days, and (c) the module reads the actual file at runtime rather than using a static profile summary. If any check fails, the module must warn and prompt before proceeding.

**Acceptance Criteria:**
1. Given the base resume file does not exist, when the module starts, then it must exit with an error identifying the missing file and indicating it cannot proceed.
2. Given the base resume exists but was last modified more than 90 days ago, when the module starts, then it must display a warning indicating the resume may be stale and prompt the user to confirm before continuing.
3. Given the module reads the base resume at runtime, when profile data is used in search queries or evaluation prompts, then the data must reflect the current file content, not a hardcoded profile summary embedded in the module.
4. Given the base resume passes all checks, when the module starts, then no onboarding warnings must be shown and the module must proceed immediately.

## L2-024: Sensitive Data Exclusion from Version Control
**Traces to:** L1-015

The repository's version control exclusion configuration must exclude all files containing personal compensation targets, contact data, application history, evaluation artifacts, and outreach content. These files must not appear in any public repository.

**Acceptance Criteria:**
1. Given the pipeline tracker file exists locally, when version control status is checked, then the file must not appear as tracked or untracked in the output.
2. Given the evaluations directory contains evaluation files, when all files are staged, then no file from that directory must be staged.
3. Given the exclusion configuration is reviewed, when the list of exclusions is compared to the required set, then all required exclusions must be present.
4. Given a file is accidentally staged despite being excluded, when version control status is checked, then the file must show as excluded if the force-add flag was not used.

---

## L2-025: Evaluation Report Storage
**Traces to:** L1-004

Each offer evaluation must be saved as a standalone file in the evaluations directory immediately after completion, using a consistent naming convention. The file must be linked from the corresponding pipeline tracker entry.

**Acceptance Criteria:**
1. Given an offer evaluation is completed, when the file is saved, then it must be named using slugified versions of the company and role names (lowercase, hyphens, no spaces) in the evaluations directory.
2. Given an evaluation file is saved, when the pipeline tracker is opened, then the corresponding row's document link column must contain a relative path to the evaluation file.
3. Given an evaluation file already exists for the same company+role, when a new evaluation is run, then the old file must be renamed with a date suffix before the new file is written.

---

## L2-026: Search Output Format Standardization
**Traces to:** L1-005, L1-014

The job search module must produce output in a consistent, machine-parseable Markdown format that can be written to the dated results file without transformation. The format must include a metadata header block, a summary statistics section, and a per-listing block for each result.

**Acceptance Criteria:**
1. Given the module completes a search, when output is written to a result file, then the file must begin with a YAML front-matter block containing: `date`, `total_results`, `boards_searched`, `new_listings`, and `seen_listings` fields.
2. Given the result file is parsed programmatically, when each listing block is read, then it must contain all required fields (`title`, `company`, `source`, `date`, `rate`, `url`, `archetype`, `score`) in a consistent format.
3. Given two consecutive search runs, when both result files are diffed, then the diff must be interpretable by the diff summary generator defined in L2-015 without additional preprocessing.
